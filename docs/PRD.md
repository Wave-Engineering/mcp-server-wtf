## 1. Problem Domain

### 1.1 Background

DevOps and SRE work involves frequent incident troubleshooting — something breaks, engineers investigate, form theories, make changes, and eventually resolve the issue. This work happens inside Claude Code sessions where the AI assistant runs commands, reads logs, and collaborates on fixes.

The troubleshooting process generates valuable knowledge: which commands were run, what was observed, what theories were formed, and which actions resolved the issue. This knowledge is exactly what's needed to write runbooks, automate fixes, and train other engineers.

### 1.2 Problem Statement

Troubleshooting context is the most perishable knowledge in DevOps. It is lost to:

- **Context compaction** — Claude Code summarizes and discards conversation history as the context window fills, destroying the detailed sequence of actions and observations
- **Session boundaries** — troubleshooting may span multiple sessions, and history from prior sessions is not carried forward
- **Human memory** — by the time someone says "let's automate that fix," the specific steps are hazy and scattered

The result: teams repeatedly troubleshoot the same classes of issues because the knowledge from prior incidents was never captured in a durable, structured form.

### 1.3 Proposed Solution

**WTF (Why That Failed)** is a flight recorder for incident troubleshooting sessions. It consists of three integrated layers:

1. **MCP Server** — a local tool server that provides journal storage and retrieval via `wtf_now`, `wtf_happened`, and `wtf_freshell` tools
2. **PostToolUse Hook** — automatically captures every tool call Claude makes during a session, creating a complete audit trail with zero manual effort
3. **Skills** — `/wtf` initiates a troubleshooting session (clears journal, enables flight recorder mode), `/wtf now` adds manual journal entries

Data is persisted to a SQLite database on disk, ensuring it survives context compaction with zero ambient token cost. A background classifier (Claude Haiku) enriches raw entries into a distilled journal, filtering noise and categorizing meaningful events — so that `wtf_happened` returns a clean, actionable timeline and auto-generated runbook skeleton.

### 1.4 Target Users

| Persona | Description | Primary Use Case |
|---------|-------------|------------------|
| **DevOps Engineer** | Uses Claude Code for infrastructure troubleshooting and incident response | Runs `/wtf` at the start of an incident, troubleshoots normally, then calls `wtf_happened` to get a distilled timeline and runbook skeleton for documentation or automation |
| **SRE / On-Call** | Responds to alerts and needs to hand off context to other engineers | Uses the WTF journal as a handoff artifact — the next engineer can read what was tried, what was observed, and what fixed it |
| **Team Lead** | Reviews incidents post-mortem and wants to improve team processes | Uses archived WTF journals to identify patterns across incidents and prioritize automation efforts |

### 1.5 Non-Goals

- **Not a monitoring or alerting system.** WTF records what happened during human-driven troubleshooting. It does not watch systems, detect anomalies, or trigger alerts. Use Grafana, Datadog, or PagerDuty for that.
- **Not a multi-incident tracker.** v1 supports one active incident at a time. Named/concurrent incidents are a v2 feature. For now, `wtf_freshell` starts a fresh session.
- **Not a replacement for proper runbook authoring.** The runbook skeleton generated by `wtf_happened` is a starting point, not a finished document. A human (or a follow-up agent session) should refine it.
- **Not a session replay tool.** WTF captures structured events, not the full conversation transcript. Use Clawback for session replay.
- **Not a remote/shared service.** WTF is local to a single machine and session. It does not sync across machines or share journals between users.

## 2. Constraints

### 2.1 Technical Constraints

| ID | Constraint | Rationale |
|----|-----------|-----------|
| CT-01 | Must be implemented as an MCP server using the `@modelcontextprotocol/sdk` | Lab exercise requirement; also the correct architecture for extending Claude Code capabilities |
| CT-02 | Runtime must be Bun + TypeScript | The lab scaffold uses Bun; `bun:sqlite` provides zero-dependency SQLite access |
| CT-03 | All persistent state must live on disk (SQLite), not in conversation context | Must survive context compaction — the core design requirement |
| CT-04 | Zero ambient token cost when idle | The journal must not consume context window tokens unless explicitly queried via `wtf_happened` |
| CT-05 | Background classifier must use Claude Haiku via AWS Bedrock (`@anthropic-ai/bedrock-sdk`) | Direct API call from TypeScript — native structured outputs, no subprocess spawning, CI-testable via project CICD variables. Fallback: `claude --model haiku --print` for users without Bedrock access |
| CT-06 | Database must support concurrent read/write from MCP server + classifier | SQLite WAL mode satisfies this; two processes (MCP server + classifier) write to separate tables |

### 2.2 Product Constraints

| ID | Constraint | Rationale |
|----|-----------|-----------|
| CP-01 | Single active incident at a time | Keeps the UX simple for v1; named incidents deferred to v2 |
| CP-02 | `wtf_happened` default output must not exceed 50 lines | Minimize token impact when the user or Claude reads the summary |
| CP-03 | Installation must be a single command (`install.sh`) | Follow the claudecode-workflow install pattern; reduce onboarding friction |
| CP-04 | Must work without network access (except for the Haiku classifier) | The MCP server and hook must function offline; only the background classifier needs Bedrock API access |

## 3. Requirements (EARS Format)

Requirements follow the **EARS** (Easy Approach to Requirements Syntax) notation:

- **Ubiquitous**: The system shall [function].
- **Event-driven**: When [event], the system shall [function].
- **State-driven**: While [state], the system shall [function].
- **Optional**: Where [feature/condition], the system shall [function].
- **Unwanted**: If [unwanted condition], then the system shall [function].

### 3.1 Recording

| ID | Type | Requirement |
|----|------|-------------|
| R-01 | Ubiquitous | The system shall accept journal entries via the `wtf_now` MCP tool with `gen_type` (`crafted` or `captured`) as a required field |
| R-02 | Optional | Where `action_type` is provided (`action`, `breadcrumb`, or `theory`), the system shall store it with the raw entry |
| R-03 | Event-driven | When a Claude Code tool call completes, the PostToolUse hook shall invoke `wtf_now` with `gen_type: "captured"` and the tool's name, input, and response |
| R-04 | Ubiquitous | The system shall persist all raw entries to the `raw_entries` table in a SQLite database at `.wtf/wtf.db` |
| R-05 | Ubiquitous | The system shall use SQLite WAL mode for concurrent read/write access |
| R-06 | Ubiquitous | Each raw entry shall include a timestamp generated at recording time |

### 3.2 Classification

| ID | Type | Requirement |
|----|------|-------------|
| R-07 | Event-driven | When a new unclassified raw entry exists, the background classifier shall evaluate it within 30 seconds |
| R-08 | Ubiquitous | The classifier shall assign each entry an `action_type` of `action`, `breadcrumb`, `theory`, or `noise` using structured output constraints |
| R-09 | Ubiquitous | The classifier shall use Claude Haiku via `@anthropic-ai/bedrock-sdk`, using structured output constraints, few-shot examples, and direct labeling (no chain-of-thought). Fallback: `claude --model haiku --print` if Bedrock is unavailable |
| R-10 | Ubiquitous | The classifier shall bias toward keeping entries — when confidence is low, the entry shall be classified as the most likely non-noise category |
| R-11 | Ubiquitous | Entries classified as `noise` shall be soft-deleted (flagged in the database, not removed) |
| R-12 | Ubiquitous | Classified entries shall be written to the `distilled_entries` table with a one-line summary and the assigned `action_type` |

### 3.3 Retrieval

| ID | Type | Requirement |
|----|------|-------------|
| R-13 | Ubiquitous | The `wtf_happened` tool shall return a Markdown-formatted timeline of the current incident from distilled entries |
| R-14 | Ubiquitous | The `wtf_happened` default response shall not exceed 50 lines |
| R-15 | Optional | Where `detail: "full"` is specified, `wtf_happened` shall return all distilled entries without truncation |
| R-16 | Event-driven | When `wtf_happened` is called, the system shall write a runbook skeleton to `.wtf/runbook.md` |
| R-17 | State-driven | While distilled entries are unavailable (classifier has not yet processed raw entries), `wtf_happened` shall fall back to reading raw entries directly and note the fallback in its output |

### 3.4 Lifecycle

| ID | Type | Requirement |
|----|------|-------------|
| R-18 | Event-driven | When `wtf_freshell` is called, the system shall archive the current incident and reset active state |
| R-19 | Ubiquitous | The system shall support exactly one active incident at a time |
| R-20 | Unwanted | If `wtf_now` is called with no active incident, the system shall create one automatically |

### 3.5 Integration

| ID | Type | Requirement |
|----|------|-------------|
| R-21 | Ubiquitous | The system shall register as an MCP server using stdio transport via `claude mcp add` |
| R-22 | Ubiquitous | The `/wtf` skill shall call `wtf_freshell`, then instruct Claude to enter flight recorder mode with a bias toward recording |
| R-23 | Ubiquitous | The `/wtf now` skill shall accept freeform text and invoke `wtf_now` with `gen_type: "crafted"` |

### 3.6 Resilience

| ID | Type | Requirement |
|----|------|-------------|
| R-25 | Ubiquitous | All journal data shall persist to disk and survive context compaction |
| R-26 | Ubiquitous | The system shall have zero ambient token cost when not actively being queried |
| R-27 | Unwanted | If the background classifier process crashes, the MCP server shall continue to function (recording and retrieval from raw entries) |

| ID | Type | Requirement |

### 4.2 Primary Flow: Troubleshooting Session

1. Engineer encounters an issue and runs `/wtf` in Claude Code
2. The `/wtf` skill calls `wtf_freshell` (archiving any prior incident), then injects behavioral instructions telling Claude to enter flight recorder mode
3. Claude and the engineer troubleshoot normally — running commands, reading logs, discussing theories
4. On every tool call, the PostToolUse hook fires and sends the tool name, input, and response to `wtf_now` with `gen_type: "captured"`
5. When Claude or the engineer has a meaningful finding, Claude calls `wtf_now` with `gen_type: "crafted"` and an appropriate `action_type`
6. The engineer can also use `/wtf now "the pool size env var was wrong"` for explicit manual entries
7. In the background, the classifier picks up new raw entries, categorizes them, and writes summaries to the distilled journal
8. At any point, the engineer calls `wtf_happened` to get a clean timeline and runbook skeleton

### 4.3 Classification Flow: Background Enrichment

1. The MCP server spawns the classifier as a child process on startup
2. The classifier polls `raw_entries` for rows with no corresponding `distilled_entries` record
3. For each unclassified entry, the classifier calls Claude Haiku via the `@anthropic-ai/bedrock-sdk` with the entry text, few-shot examples, and structured output constraints (falls back to `claude --model haiku --print` if Bedrock is unavailable)
4. Haiku returns a category (`action`, `breadcrumb`, `theory`, or `noise`) and a one-line summary
5. The classifier writes the result to `distilled_entries`, referencing the original `raw_id`
6. Entries classified as `noise` are written with `is_noise = true` — soft-deleted, not removed
7. If the classifier crashes, the MCP server continues to function; `wtf_happened` falls back to raw entries

### 4.4 Retrieval Flow: wtf_happened

1. `wtf_happened` queries `distilled_entries` for the active incident, filtering out noise
2. If no distilled entries exist yet (classifier hasn't caught up), it falls back to `raw_entries` and notes this in the output
3. The tool returns a concise Markdown timeline (capped at 50 lines by default) showing timestamped, categorized events
4. Simultaneously, it writes a full runbook skeleton to `.wtf/runbook.md` containing: problem description (from breadcrumbs), root cause (from theories), resolution steps (from actions), and verification steps
5. The in-context response includes the timeline plus a pointer to the runbook file path

### 4.5 Lifecycle Flow: Clear and Archive

1. `wtf_freshell` is called (either directly or via the `/wtf` skill)
2. The active incident's status is set to `archived` with an `ended_at` timestamp
3. A new incident row is created with status `active`
4. Raw and distilled entries from the prior incident remain in the database (queryable by incident ID) but are not returned by `wtf_happened`

## 5. Detailed Design

### 5.1 Data Model

The database lives at `.wtf/wtf.db` (relative to the project root where CLAUDE.md exists). SQLite is opened in WAL mode on first connection.

**Table: `incidents`**

```sql
CREATE TABLE incidents (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT,
    started_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%f', 'now')),
    ended_at    TEXT,
    status      TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'archived'))
);
```

**Table: `raw_entries`**

```sql
CREATE TABLE raw_entries (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    incident_id    INTEGER NOT NULL REFERENCES incidents(id),
    ts             TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%f', 'now')),
    gen_type       TEXT NOT NULL CHECK (gen_type IN ('crafted', 'captured')),
    action_type    TEXT CHECK (action_type IN ('action', 'breadcrumb', 'theory') OR action_type IS NULL),
    text           TEXT,
    tool_name      TEXT,
    tool_input     TEXT,    -- JSON string
    tool_response  TEXT,    -- JSON string
    tool_use_id    TEXT,
    agent_id       TEXT,
    agent_type     TEXT,
    session_id     TEXT
);

CREATE INDEX idx_raw_incident ON raw_entries(incident_id);
```

**Table: `distilled_entries`**

```sql
CREATE TABLE distilled_entries (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    raw_id          INTEGER NOT NULL REFERENCES raw_entries(id),
    incident_id     INTEGER NOT NULL REFERENCES incidents(id),
    ts              TEXT NOT NULL,
    action_type     TEXT NOT NULL CHECK (action_type IN ('action', 'breadcrumb', 'theory', 'noise')),
    summary         TEXT NOT NULL,
    is_noise        INTEGER NOT NULL DEFAULT 0,
    classified_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%f', 'now'))
);

CREATE INDEX idx_distilled_incident ON distilled_entries(incident_id);
CREATE INDEX idx_distilled_not_noise ON distilled_entries(incident_id) WHERE is_noise = 0;
```

### 5.2 API Surface (MCP Tools)

#### Tool: `wtf_now`

Records a journal entry into the raw journal.

```typescript
{
  name: "wtf_now",
  description: "Record a journal entry in the WTF flight recorder. Use gen_type 'crafted' for intentional entries, 'captured' for automatic hook entries.",
  inputSchema: {
    type: "object",
    properties: {
      gen_type:       { type: "string", enum: ["crafted", "captured"], description: "How this entry was generated" },
      action_type:    { type: "string", enum: ["action", "breadcrumb", "theory"], description: "Category (optional for captured)" },
      text:           { type: "string", description: "Freeform description of what happened" },
      tool_name:      { type: "string", description: "Tool name (for captured entries)" },
      tool_input:     { type: "string", description: "Tool input as JSON string (for captured entries)" },
      tool_response:  { type: "string", description: "Tool response as JSON string (for captured entries)" },
      tool_use_id:    { type: "string", description: "Unique tool use ID (for captured entries)" },
      agent_id:       { type: "string", description: "Sub-agent ID if applicable" },
      agent_type:     { type: "string", description: "Agent type if applicable" },
      session_id:     { type: "string", description: "Session identifier" }
    },
    required: ["gen_type"]
  }
}
```

**Returns:** `{ ok: true, entry_id: <number> }`

**Error cases:**
- Invalid `gen_type` → error with message
- Database write failure → error with message

#### Tool: `wtf_happened`

Returns a distilled timeline of the current incident.

```typescript
{
  name: "wtf_happened",
  description: "Get a distilled timeline of the current troubleshooting incident. Writes a full runbook to .wtf/runbook.md.",
  inputSchema: {
    type: "object",
    properties: {
      detail: {
        type: "string",
        enum: ["summary", "full"],
        description: "summary (default, max 50 lines) or full (all entries)"

```

**Returns (summary mode):**
```markdown

**Duration:** 45 min | **Entries:** 127 raw, 34 distilled | **Status:** active

### Timeline
1. [12:03] BREADCRUMB — Health endpoint returning 503s intermittently

```

**Returns (full mode):** Same format without the 50-line cap.

**Fallback:** If no distilled entries exist, reads from `raw_entries` and prepends:
```
> Note: Classifier has not processed entries yet. Showing raw journal.
```

### 5.3 PostToolUse Hook

The hook is a shell script at `scripts/hooks/wtf-post-tool-use.sh`. It reads the PostToolUse JSON payload from stdin and invokes `wtf_now` via the MCP server.

**Input (from Claude Code, on stdin):**
```json
{
  "session_id": "abc123",
  "hook_event_name": "PostToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "curl http://localhost:8080/health" },
  "tool_response": { "output": "{\"status\":\"degraded\"}" },
  "tool_use_id": "toolu_01ABC123",
  "cwd": "/home/user/project",
  "agent_id": "optional",
  "agent_type": "optional"
}
```

**Hook behavior:**
1. Read JSON from stdin
2. Extract fields: `tool_name`, `tool_input`, `tool_response`, `tool_use_id`, `session_id`, `agent_id`, `agent_type`
3. Truncate `tool_input` and `tool_response` to 4KB each (prevent oversized entries from commands with large output)
4. Write a JSON line to `.wtf/hook-queue.jsonl` (append-only)
5. The MCP server polls this file and ingests entries into `raw_entries`
6. Exit 0 immediately (hook must not block Claude)

**Why a queue file instead of direct DB write?** The hook script runs as a child process of Claude Code, not the MCP server. It doesn't have the DB connection. A lightweight append to a JSONL file is atomic (single write) and fast. The MCP server ingests from this queue on its next poll cycle.

**Hook registration (in `~/.claude/settings.json`):**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "type": "command",
        "command": "/path/to/scripts/hooks/wtf-post-tool-use.sh"
      }
    ]
  }
}
```

### 5.4 Background Classifier

The classifier is a long-running process spawned by the MCP server on startup. It runs as a loop:

```
while true:
    1. Query raw_entries for rows not yet in distilled_entries
    2. If none, sleep 5 seconds and retry
    3. For each unclassified entry:
       a. Build prompt with entry text + few-shot examples
       b. Call Haiku via Bedrock SDK with structured output (JSON Schema constraint)
       c. Parse response: { action_type, summary }
       d. Insert into distilled_entries
       e. If action_type == "noise", set is_noise = 1
    4. Sleep 2 seconds between entries (rate limiting)
```

**Bedrock integration:**

```typescript
import AnthropicBedrock from "@anthropic-ai/bedrock-sdk";

const client = new AnthropicBedrock({
  awsRegion: "us-east-1",
  // AWS credentials resolved via standard SDK chain:
  // env vars → shared credentials file → instance role
});

const response = await client.messages.create({
  model: "us.anthropic.claude-haiku-4-5-20251001",
  max_tokens: 256,
  messages: [{ role: "user", content: prompt }],
  // Structured output via tool_use forces valid JSON schema
});
```

**Configuration:** AWS credentials are resolved in this order:
1. Environment variables (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`) — used in CI via project CICD variables
2. AWS credentials file with `--profile dev-deploy-bot` — used in local development
3. **Fallback:** if Bedrock is unavailable, spawn `claude --model haiku --print` as a subprocess

**Classifier prompt structure** (see Appendix A for full prompt):
- System instruction: "Classify this troubleshooting event"
- Four categories with definitions
- 4-5 few-shot examples with contrastive pairs
- Structured output enforced via JSON Schema constraint (Bedrock supports this natively)
- Instruction to bias toward keeping when uncertain

**Crash resilience:** If the classifier process exits, the MCP server logs a warning and continues. `wtf_happened` falls back to raw entries. The classifier can be restarted by re-running `install.sh` or manually.

### 5.5 Skills

#### Skill: `/wtf`

Location: `skills/wtf/SKILL.md`

Behavior:
1. Call `wtf_freshell` to archive any prior incident and start fresh
2. Prompt user for an optional incident title/description
3. Inject behavioral instructions to Claude:

> You are now in **WTF flight recorder mode**. A background journal is capturing every tool call automatically. In addition to that automatic capture, you should:
>
> - Call `wtf_now` with `gen_type: "crafted"` and `action_type: "breadcrumb"` when you observe something diagnostically significant
> - Call `wtf_now` with `gen_type: "crafted"` and `action_type: "theory"` when you form a hypothesis about the root cause
> - Call `wtf_now` with `gen_type: "crafted"` and `action_type: "action"` when you take a corrective action (not exploratory commands)
>
> **Bias toward recording.** When in doubt, record it. A noisy journal can be distilled; a silent journal cannot be recovered.
>
> At any time, the user may call `wtf_happened` to see the current timeline, or `/wtf now` to add their own observations.

#### Skill: `/wtf now`

Location: `skills/wtf-now/SKILL.md`

Behavior:
1. Parse the user's argument as freeform text
2. Call `wtf_now` with `gen_type: "crafted"` and `text: <user's text>`
3. Optionally infer `action_type` from the text (or leave it for the classifier)

### 5.7 Technology Choices

| Component | Technology | Rationale |
|-----------|-----------|-----------|
| Runtime | Bun | Lab scaffold uses Bun; fast startup (~50ms); native TypeScript |
| MCP SDK | `@modelcontextprotocol/sdk` | Standard MCP server library |
| Database | SQLite3 (WAL mode) | Concurrent read/write; zero-config; file-based; survives compaction |
| SQLite binding | `bun:sqlite` | Built into Bun; no extra dependencies |
| Classifier | `@anthropic-ai/bedrock-sdk` | Native structured outputs; direct API call from TypeScript; CI-testable via CICD variables. Fallback: `claude --model haiku --print` |
| Test framework | `bun test` | Built into Bun; Jest-compatible API |
| Hook transport | JSONL queue file | Decouples hook (short-lived process) from MCP server (long-lived); atomic append |

### 5.8 Open Questions

1. **Queue file vs. direct MCP call from hook:** The current design uses a JSONL queue file because the hook script can't call MCP tools directly. An alternative is to have the hook script write directly to the SQLite DB. This would eliminate the queue but create a third writer (hook + MCP + classifier). Decided: queue file is simpler and avoids coordination concerns.

2. **Classifier structured output via Bedrock SDK:** The `@anthropic-ai/sdk` supports structured output via tool_use or JSON mode. Decided: use the SDK directly with Bedrock transport. JSON Schema constraints are enforced natively. The CLI fallback (`claude --model haiku --print`) does not guarantee structured output — if falling back, the worker must parse and validate freeform JSON.

3. **Hook queue ingestion frequency:** The MCP server polls `hook-queue.jsonl` — how often? Too fast wastes CPU; too slow means entries are delayed. Starting at 2-second poll interval, tunable.

**Implementation Steps:**

1. **Create `scripts/hooks/wtf-post-tool-use.sh`:**
   ```bash
   #!/usr/bin/env bash
   # WTF PostToolUse hook — captures tool calls to the flight recorder queue
   ```
   - Read JSON from stdin: `INPUT=$(cat)`
   - Extract fields with `jq`: `tool_name`, `tool_input`, `tool_response`, `tool_use_id`, `session_id`, `agent_id`, `agent_type`
   - Truncate `tool_input` and `tool_response` to 4096 bytes each: pipe through `head -c 4096`
   - Determine queue path: `$CLAUDE_PROJECT_DIR/.wtf/hook-queue.jsonl`
   - Ensure `.wtf/` directory exists: `mkdir -p "$CLAUDE_PROJECT_DIR/.wtf"`
   - Build output JSON with `jq -nc` and append to queue file
   - Exit 0 immediately
   - Make executable: `chmod +x`

2. **Create `mcp-server/queue.ts`:**
   - Export a `startQueueIngestion(db, queuePath, intervalMs)` function
   - Uses `setInterval` to poll the queue file
   - On each tick: read `hook-queue.jsonl`, parse each line, insert into `raw_entries` with `gen_type: "captured"`, truncate the file
   - Use file locking or atomic rename to avoid race conditions with the hook writer:
     a. Rename `hook-queue.jsonl` to `hook-queue.processing.jsonl`
     b. Read and process the renamed file
     c. Delete the renamed file
   - Log errors but do not crash

3. **Create `mcp-server/tests/queue.test.ts`:**

   - Test that multiple lines are ingested in order
   - Test that the queue file is consumed (deleted after processing)

   - Export a `CLASSIFIER_SCHEMA` JSON Schema object for structured output:
     ```typescript
     export const CLASSIFIER_SCHEMA = {
       type: "object",
       properties: {
         action_type: { type: "string", enum: ["action", "breadcrumb", "theory", "noise"] },
         summary: { type: "string", description: "One-line summary (empty for noise)" }
       },
       required: ["action_type", "summary"]
     };
     ```

4. **Create `mcp-server/classifier/worker.ts`:**
   - Export a `startClassifier(db, options)` function
   - Options: `{ pollIntervalMs: number, rateLimitMs: number }`
   - Defaults: `pollIntervalMs: 5000`, `rateLimitMs: 2000`
   - The loop:
     a. Query: `SELECT r.* FROM raw_entries r LEFT JOIN distilled_entries d ON r.id = d.raw_id WHERE d.id IS NULL ORDER BY r.id ASC LIMIT 1`
     b. If no result, sleep `pollIntervalMs`
     c. Build prompt from the entry
     d. Call Haiku via Bedrock SDK with structured output (tool_use pattern or JSON mode):
        ```typescript
        const response = await client.messages.create({

          max_tokens: 256,

        ```
     e. Parse response, validate `action_type` is one of the four categories
     f. Insert into `distilled_entries`:
        - `raw_id`: the raw entry's ID
        - `incident_id`: from the raw entry
        - `ts`: from the raw entry
        - `action_type`: from Haiku response
        - `summary`: from Haiku response
        - `is_noise`: 1 if `action_type === "noise"`, else 0

   - Wrap the entire loop in try/catch — log errors, continue
   - If Bedrock call fails (auth, network), attempt CLI fallback before skipping

### Appendix A: Classifier Prompt

The following is the prompt template used by the background classifier. The `{{entry}}` placeholder is replaced with the raw entry's content.

```
You classify troubleshooting session events into exactly one category.
Respond with JSON only: {"action_type": "<category>", "summary": "<one-line summary>"}
For noise entries, set summary to "".

Categories:
- action: A meaningful action taken to diagnose or fix the issue (ran a command that changed state, modified config, restarted a service, applied a patch)
- breadcrumb: An observation or data point that provides diagnostic context (error message seen, metric value noted, log entry found, system state observed)
- theory: Reasoning, hypothesis, or analysis about the problem cause or mechanism
- noise: Navigation, repeated lookups, file listing without findings, or content with no diagnostic value

Classification rules:
- When uncertain between noise and any other category, choose the non-noise category
- If an entry contains both an action and a theory ("restarted because I think it's leaking"), classify as action
- Captured tool calls that produce errors or unexpected output are breadcrumbs, not noise
- File reads and greps are noise UNLESS the output contains diagnostic information

<examples>
<example>
Entry: {"tool_name": "Bash", "tool_input": {"command": "kubectl rollout restart deployment/api-server"}, "tool_response": {"output": "deployment.apps/api-server restarted"}}
{"action_type": "action", "summary": "Restarted api-server deployment via kubectl rollout restart"}
</example>

<example>
Entry: {"tool_name": "Bash", "tool_input": {"command": "curl -s http://localhost:8080/health | jq ."}, "tool_response": {"output": "{\"status\":\"degraded\",\"db\":\"timeout\"}"}}
{"action_type": "breadcrumb", "summary": "Health endpoint reports degraded status with database timeout"}
</example>

<example>
Entry: {"gen_type": "crafted", "text": "I think the connection pool is exhausting because the idle timeout is set to 0 — connections never close"}
{"action_type": "theory", "summary": "Hypothesis: connection pool exhaustion due to zero idle timeout setting"}
</example>

<example>
Entry: {"tool_name": "Bash", "tool_input": {"command": "ls /var/log/"}, "tool_response": {"output": "auth.log\ndpkg.log\nsyslog\n..."}}
{"action_type": "noise", "summary": ""}
</example>

<example>
Entry: {"tool_name": "Bash", "tool_input": {"command": "grep OOM /var/log/syslog"}, "tool_response": {"output": "Mar 30 11:42:03 host kernel: [12345.678] Out of memory: Killed process 5432 (java)"}}
{"action_type": "breadcrumb", "summary": "Found OOM killer event in syslog — Java process 5432 was killed"}
</example>
</examples>

Classify this entry:
{{entry}}
```

### Appendix B: Runbook Template

```markdown
# Runbook: [incident title]

## Resolution Steps
[Ordered list of action entries — what was done to fix it]

2. [action 2]
[If no actions recorded: "No corrective actions were recorded during this incident."]

## Full Timeline

```
